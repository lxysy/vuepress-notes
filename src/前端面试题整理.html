    # 前端面试题整理（含答案）

    本文档整理了金山云外包、盛天网络和东高科技、武汉中地云申、达昌技术、百安居、万云科技等公司的前端面试题及详细回答。

    # 一、金山云外包一面

    # 1. 防抖和节流

    防抖（Debounce）：

    • 在事件被触发 n 秒后再执行回调，如果在这 n 秒内又被触发，则重新计时
    • 适用场景：搜索框输入、窗口 resize、表单验证
    • 实现：
    function debounce(fn, delay) {
  let timer = null;
  return function (...args) {
    if (timer) clearTimeout(timer);
    timer = setTimeout(() => {
      fn.apply(this, args);
    }, delay);
  };
}

    节流（Throttle）：

    • 规定在一个单位时间内，只能触发一次函数。如果这个单位时间内触发多次函数，只有一次生效
    • 适用场景：滚动加载、按钮点击、鼠标移动
    • 实现（时间戳版）：
    function throttle(fn, interval) {
  let lastTime = 0;
  return function (...args) {
    const now = Date.now();
    if (now - lastTime >= interval) {
      lastTime = now;
      fn.apply(this, args);
    }
  };
}

    区别：

    • 防抖：事件停止触发后一段时间才执行（重新计时）
    • 节流：事件持续触发时，每隔一段时间执行一次（有节奏地执行）

    # 2. 用递归实现阶乘（如传入5，返回54321）

    function factorial(n) {
  // 基准条件
  if (n <= 1) return 1;
  // 递归调用
  return n * factorial(n - 1);
}

console.log(factorial(5)); // 120

    尾递归优化版：

    function factorial(n, result = 1) {
  if (n <= 1) return result;
  return factorial(n - 1, n * result);
}

    # 3. 箭头函数和普通函数的区别

    特性 箭头函数 普通函数
    this 指向 继承外层作用域的 this，不可改变 调用时确定，可通过 call/apply/bind 改变
    arguments 没有自己的 arguments，可用 rest 参数 有自己的 arguments 对象
    构造函数 不能作为构造函数，不能用 new 可以作为构造函数
    prototype 没有 prototype 属性 有 prototype 属性
    返回值 单条语句可隐式返回 需要显式 return
    适用场景 回调函数、数组方法 对象方法、构造函数

    注意：箭头函数不适合定义对象方法（因为 this 不指向对象本身）

    # 4. HTTPS 和 HTTP 的区别

    特性 HTTP HTTPS
    安全性 明文传输，不安全 SSL/TLS 加密传输，安全
    默认端口 80 443
    证书 不需要 需要 SSL 证书（CA 颁发）
    性能 相对较快 需要加密解密，开销稍大
    SEO 无优势 搜索引擎优先收录

    HTTPS 加密过程：

    1. 客户端向服务器发送 HTTPS 请求
    2. 服务器返回公钥证书
    3. 客户端验证证书合法性
    4. 客户端生成对称密钥，用公钥加密后发送给服务器
    5. 服务器用私钥解密，获得对称密钥
    6. 双方使用对称密钥加密通信

    # 5. 前端缓存

    前端缓存分为强缓存和协商缓存：

    强缓存（不发请求到服务器）：

    • Cache-Control: max-age=xxx（HTTP/1.1，单位秒）
    • Expires: 具体时间（HTTP/1.0，绝对时间）
    • 状态码：200 (from memory cache) 或 200 (from disk cache)

    协商缓存（发请求到服务器确认）：

    • Last-Modified / If-Modified-Since（基于文件修改时间）
    • ETag / If-None-Match（基于文件内容哈希，优先级更高）
    • 状态码：304 Not Modified

    缓存优先级： Service Worker > Memory Cache > Disk Cache > Push Cache > 网络请求

    # 【补充】前端缓存的实际应用场景与价值

    作为前端开发者，为什么要关心缓存？

    缓存是前端性能优化中最直接、最有效的手段之一。合理配置缓存可以减少网络请求、降低服务器压力、提升页面加载速度和用户体验。前端开发者需要与后端/运维协作，但更需要理解原理以便排查问题和提出优化方案。

    1. 强缓存的实际应用场景

    场景一：静态资源长期缓存（JS/CSS/图片/字体）

    前端构建工具（Webpack/Vite）打包后的资源文件名包含哈希值（如 main.a2f1b3c.js），文件内容变化时哈希变化，URL 也变化。这种资源可以设置长期强缓存：

    # Nginx 配置示例
location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2|ttf|eot)$ {
    expires 1y;                    # 缓存一年
    add_header Cache-Control "public, immutable";
}

    效果：用户首次访问后，这些资源在一年内直接从本地磁盘缓存读取，零网络请求，页面秒开。

    前端工作：

    • 确保构建产物启用 [contenthash] 或 [hash] 命名
    • 在 HTML 中引用带哈希的资源（通过构建工具自动注入）

    场景二：第三方库 CDN 缓存

    使用 CDN 加载 Vue/React/jQuery 等库时，利用 CDN 的缓存分发能力：

    <script src="https://cdn.jsdelivr.net/npm/vue@3.2.0/dist/vue.global.js"></script>

    CDN 节点已对这些资源做了强缓存，用户可能从距离最近的节点毫秒级获取。

    前端工作：

    • 评估是否使用 CDN（公共库适合，业务代码不适合）
    • 配置 integrity 属性防止 CDN 篡改：
    <script src="https://cdn.example.com/lib.js" 
        integrity="sha384-xxx..." 
        crossorigin="anonymous"></script>

    2. 协商缓存的实际应用场景

    场景三：HTML 入口文件使用协商缓存

    HTML 文件不能做强缓存（否则无法更新引用的 JS/CSS），但需要避免重复下载未修改的内容：

    # Nginx 配置
location /index.html {
    expires -1;                    # 禁用强缓存
    add_header Cache-Control "no-cache";  # 强制协商缓存
}

    工作流程：

    1. 用户每次访问，浏览器都请求 HTML
    2. 服务端比对 ETag 或 Last-Modified
    3. 如果 HTML 未修改 → 返回 304 Not Modified，浏览器使用缓存（仅消耗几百字节）
    4. 如果 HTML 修改了（如引用了新版本的 JS）→ 返回 200 和新内容

    前端价值：

    • 保证用户总能获取最新资源引用关系
    • 避免全量下载 HTML（通常几 KB 到几十 KB）
    • 是现代化构建（hash 文件名）+ 协商缓存 HTML 的标准实践

    场景四：API 接口缓存

    不常变化的数据接口（如配置信息、字典数据、商品分类）可设置短时间强缓存 + 协商缓存：

    // 前端请求时主动控制缓存（axios 示例）
axios.get('/api/config', {
  headers: {
    'Cache-Control': 'max-age=3600'  // 建议浏览器缓存1小时
  }
});

    或者由后端返回缓存头：

    HTTP/1.1 200 OK
Cache-Control: public, max-age=3600
ETag: "config-v2"

    前端工作：

    • 与后端约定哪些接口可缓存、缓存多久
    • 对敏感数据（用户信息、订单）设置 Cache-Control: no-store

    3. 缓存策略在实际项目中的典型配置

    现代化单页应用（SPA）的推荐缓存策略：

    资源类型 缓存策略 原因
    index.html 协商缓存（no-cache） 必须获取最新资源引用
    *.js / *.css（带 hash） 强缓存 1年（immutable） 内容变化 URL 变，可永久缓存
    图片/字体/视频 强缓存 1年 不常变化，体积大
    API 数据（静态） 强缓存 1小时 + ETag 减少重复请求
    API 数据（动态） 协商缓存或不缓存 保证数据实时性

    4. 前端开发中的缓存问题排查

    常见问题 1：更新了代码，用户还是看到旧页面

    • 原因：HTML 或 JS 被强缓存
    • 排查：打开 Network 面板，看状态码是 200 (from disk cache) 还是 304
    • 解决：
      • 确保 HTML 使用协商缓存而非强缓存
      • 确保 JS/CSS 文件名包含 hash
      • 紧急修复时修改 HTML 的查询参数强制刷新：<script src="app.js?v=2">

    常见问题 2：调试时缓存干扰开发

    • 解决：
      • Chrome DevTools → Network → 勾选 Disable cache
      • 按 Ctrl+Shift+R（Windows）或 Cmd+Shift+R（Mac）强制刷新
      • 在 URL 后加随机数：?t=${Date.now()}

    常见问题 3：缓存优先级导致 Service Worker 缓存不更新

    • 原因：Service Worker 拦截请求后，可能优先返回旧缓存
    • 排查：Application → Service Workers → 查看缓存内容
    • 解决：更新 SW 脚本，在 activate 阶段清理旧缓存（见 Service Worker 章节）

    5. 前端可以主动控制的缓存

    除了依赖 HTTP 缓存头，前端还可以通过以下方式管理缓存：

    LocalStorage / SessionStorage：

    // 缓存不常变化的配置数据（注意 5MB 限制）
const config = localStorage.getItem('app_config');
if (config) {
  return JSON.parse(config);  // 使用本地缓存
}

const freshConfig = await fetch('/api/config').then(r => r.json());
localStorage.setItem('app_config', JSON.stringify(freshConfig));

    IndexedDB：

    // 缓存大量结构化数据（如离线可用的文档、图片）
const db = await openDB('AppCache', 1, {
  upgrade(db) {
    db.createObjectStore('documents', { keyPath: 'id' });
  }
});

// 先读缓存，再请求更新（Stale While Revalidate 的前端实现）
const cached = await db.get('documents', docId);
if (cached) render(cached);  // 立即显示

const fresh = await fetch(`/api/doc/${docId}`).then(r => r.json());
await db.put('documents', fresh);
render(fresh);  // 更新为最新

    Cache API（配合 Service Worker）：

    // 在 Service Worker 中使用，精确控制缓存内容
caches.open('api-cache-v1').then(cache => {
  cache.add('/api/user/profile');      // 缓存请求
  cache.match('/api/user/profile');    // 读取缓存
  cache.delete('/api/user/profile');   // 删除缓存
});

    6. 缓存的价值总结

    指标 无缓存 合理缓存后 提升效果
    页面二次加载时间 2-5 秒 0.5-1 秒 提升 70-80%
    服务器带宽消耗 100% 20-30% 节省 70%+
    用户流量消耗 高 低 移动端尤为重要
    服务器 QPS 压力 高 低 支撑更多并发用户

    前端工程师的核心价值：

    1. 设计合理的构建产物命名策略（hash），让强缓存成为可能
    2. 与后端协商 API 缓存策略，减少不必要的数据请求
    3. 使用 Service Worker / IndexedDB 实现离线可用和秒开体验
    4. 能够快速定位和解决用户反馈的"看到旧页面"问题

    # 7. Vuex

    Vuex 是 Vue 的状态管理库，采用集中式存储管理应用的所有组件的状态。

    核心概念：

    • State：存储应用的状态数据
    • Getters：基于 state 的计算属性
    • Mutations：修改 state 的唯一途径（必须同步）
    • Actions：提交 mutations，可包含异步操作
    • Modules：将 store 分割成模块

    流程： State → Vue Components → Actions → Mutations → State

    # 8. 原型和原型链

    原型（Prototype）：

    • 每个函数都有一个 prototype 属性，指向原型对象
    • 原型对象默认有一个 constructor 属性指回构造函数
    • 实例通过 __proto__ 访问原型

    原型链（Prototype Chain）：

    • 当访问对象的属性时，先在自身查找，找不到就沿 __proto__ 向上查找
    • 直到找到 Object.prototype，其 __proto__ 为 null
    • 这条查找路径就是原型链
    // 示例
function Person() {}
const p = new Person();
p.__proto__ === Person.prototype; // true
Person.prototype.__proto__ === Object.prototype; // true
Object.prototype.__proto__ === null; // true

    # 9. 异步任务分为哪些

    JavaScript 的异步任务分为：

    宏任务（Macrotask）：

    • setTimeout / setInterval
    • setImmediate（Node.js）
    • I/O 操作
    • UI 渲染
    • requestAnimationFrame

    微任务（Microtask）：

    • Promise.then / catch / finally
    • MutationObserver
    • queueMicrotask
    • process.nextTick（Node.js，优先级最高）

    执行顺序：同步代码 → 微任务队列清空 → 宏任务 → 微任务队列清空 → 下一个宏任务...

    # 【补充】MutationObserver 与 queueMicrotask 详解

    1. MutationObserver

    是什么： MutationObserver 是浏览器提供的 DOM 变动观察器，用于异步监听 DOM 树的变化（节点增删、属性修改、文本内容变化）。它是微任务的一种实现机制。

    核心特点：

    • 异步执行：DOM 变化不会立即触发回调，而是等待当前宏任务执行完毕后，在微任务阶段批量处理
    • 批量合并：多次 DOM 变化会合并为一次回调（通过 takeRecords 获取变动列表）
    • 性能友好：相比已废弃的 MutationEvent，不会阻塞渲染线程

    常见使用场景：

    // 场景 1：监听 DOM 变化执行后续逻辑（如自动滚动到底部）
const observer = new MutationObserver((mutations, obs) => {
  // mutations 是 DOM 变动数组
  mutations.forEach((mutation) => {
    if (mutation.type === 'childList') {
      console.log('子节点变化:', mutation.addedNodes, mutation.removedNodes);
    }
    if (mutation.type === 'attributes') {
      console.log('属性变化:', mutation.attributeName, mutation.target);
    }
  });
});

// 配置观察选项
observer.observe(document.getElementById('chat-box'), {
  childList: true,        // 监听子节点增删
  attributes: true,       // 监听属性变化
  characterData: true,    // 监听文本内容变化
  subtree: true,          // 监听后代节点（不仅直接子节点）
  attributeOldValue: true,// 记录属性旧值
  characterDataOldValue: true
});

// 停止观察
observer.disconnect();

    场景 2：作为微任务调度器（历史兼容方案）

    在 queueMicrotask 出现之前，MutationObserver 是实现微任务的常用技巧：

    // 旧版浏览器中模拟微任务（现在已不推荐，用 queueMicrotask 替代）
function microTask(callback) {
  const node = document.createTextNode('');
  const observer = new MutationObserver(() => {
    callback();
    observer.disconnect();
  });
  observer.observe(node, { characterData: true });
  node.data = 'trigger'; // 触发变化
}

// 效果：callback 在当前同步代码后、下一个宏任务前执行
microTask(() => console.log('微任务执行'));
console.log('同步代码');
// 输出：同步代码 → 微任务执行

    为什么 MutationObserver 是微任务？

    浏览器设计如此：DOM 变化频繁，如果每次变化都同步触发回调，会导致大量重排重绘，阻塞 UI。将其放入微任务队列，可以：

    1. 等待当前同步代码执行完毕（避免中间状态被观察到）
    2. 批量合并多次变动（一次回调处理所有变化）
    3. 在 UI 渲染前完成（确保观察逻辑不滞后）

    2. queueMicrotask

    是什么： queueMicrotask 是浏览器和 Node.js 提供的 标准微任务调度 API，用于将回调函数放入微任务队列，在当前同步代码执行完毕后立即执行。

    语法：

    queueMicrotask(() => {
  console.log('这是一个微任务');
});

    与 setTimeout(..., 0) 的区别：

    console.log('1. 同步代码开始');

setTimeout(() => {
  console.log('4. setTimeout 宏任务');
}, 0);

queueMicrotask(() => {
  console.log('3. queueMicrotask 微任务');
});

console.log('2. 同步代码结束');

// 输出顺序：1 → 2 → 3 → 4
    特性 queueMicrotask setTimeout(..., 0)
    任务类型 微任务 宏任务
    执行时机 当前同步代码后立即执行 等待所有微任务执行完毕后，下一个宏任务阶段
    延迟 无延迟（当前事件循环末尾） 最小 4ms（HTML5 规范限制）
    使用场景 需要尽快执行但不阻塞当前任务 需要让出主线程，允许 UI 更新

    实际应用场景：

    // 场景 1：拆分大任务避免阻塞主线程（配合微任务实现伪异步）
function processLargeData(data) {
  const chunkSize = 100;
  let index = 0;
  
  function processChunk() {
    const chunk = data.slice(index, index + chunkSize);
    // 处理数据...
    
    index += chunkSize;
    if (index < data.length) {
      // 使用 queueMicrotask 而非 for 循环，让出执行权
      queueMicrotask(processChunk);
    }
  }
  
  processChunk();
}

// 场景 2：状态更新后异步执行副作用（确保 DOM 已更新）
class Component {
  update(data) {
    this.state = data;
    
    // 将副作用推入微任务，确保状态更新已完成
    queueMicrotask(() => {
      this.notifySubscribers();
      this.updateDOM();
    });
  }
}

// 场景 3：Promise 的 polyfill 实现
function MyPromise(executor) {
  // ...
  const resolve = (value) => {
    // 使用 queueMicrotask 保证 then 回调异步执行
    queueMicrotask(() => {
      this.value = value;
      this.onFulfilledCallbacks.forEach(cb => cb(this.value));
    });
  };
  executor(resolve, reject);
}

    与 Promise.then 的关系：

    // 以下三者在微任务队列中优先级相同，按入队顺序执行
Promise.resolve().then(() => console.log('Promise'));
queueMicrotask(() => console.log('queueMicrotask'));

const observer = new MutationObserver(() => console.log('MutationObserver'));
observer.observe(document.body, { childList: true });
document.body.appendChild(document.createElement('div'));

// 输出顺序（按入队顺序）：Promise → queueMicrotask → MutationObserver

    注意事项：

    // 危险：在微任务中递归创建微任务可能阻塞渲染
function dangerous() {
  queueMicrotask(() => {
    console.log('微任务');
    dangerous(); // 递归创建，宏任务永远无法执行
  });
}
// 页面将卡死，setTimeout、用户点击等宏任务无法响应

// 安全做法：需要持续执行时，间歇性让出主线程
function safe() {
  queueMicrotask(() => {
    console.log('微任务');
    if (shouldContinue) {
      setTimeout(safe, 0); // 使用宏任务让出执行权
    }
  });
}

    总结：

    API 任务类型 主要用途 现代替代方案
    Promise.then 微任务 异步流程控制、链式调用 原生支持，无需替代
    MutationObserver 微任务 DOM 变动监听、历史微任务模拟 queueMicrotask（调度）/ IntersectionObserver（可见性）
    queueMicrotask 微任务 标准微任务调度、拆分同步逻辑 原生 API，最推荐
    process.nextTick 微任务（Node.js） Node.js 中最高优先级异步操作 queueMicrotask（跨平台统一）

    理解这些微任务机制，对于分析事件循环输出、实现自定义 Promise、优化长任务拆分至关重要。

    2. 预加载（Preload / Prefetch / Preconnect）

    浏览器提供的 <link rel="..."> 资源提示机制，提前获取或准备后续需要的资源。

    属性 作用 优先级 使用场景

    dns-prefetch 提前解析 DNS 低 多域名资源预解析
    prerender 后台渲染整个页面 最低 用户极可能访问的下一页

<!-- 4. DNS-Prefetch：仅解析 DNS，比 preconnect 更轻量 -->

<!-- 5. Prerender：后台完整渲染页面（资源消耗大，谨慎使用） -->

    3. Web Worker

    Web Worker 允许在浏览器后台线程中运行 JavaScript，不阻塞主线程（UI 线程）。

    适用场景：

    • 大数据排序/过滤/聚合（Excel 数据处理）
    • 图像处理（滤镜、压缩、格式转换）
    • 复杂数学计算（加密解密、科学计算）
    • 文本分析（代码高亮、Markdown 渲染）
    • 实时数据解析（WebSocket 大数据流）

    Worker 类型：

    类型 特点 使用场景
    Dedicated Worker 专属页面，一对一通信 大部分场景
    Shared Worker 多页面/iframe 共享 跨 Tab 数据同步
    Service Worker 代理网络请求，可离线 PWA、离线缓存

    基础用法：

    // main.js（主线程）
const worker = new Worker('worker.js');

// 发送数据给 Worker（结构化克隆算法，可传大部分数据类型）
worker.postMessage({ type: 'SORT', data: largeArray });

// 接收 Worker 结果
worker.onmessage = (e) => {
  console.log('排序完成:', e.data.result);
  // 更新 UI
};

worker.onerror = (err) => {
  console.error('Worker 错误:', err.message);
};

// 不再需要时终止 Worker（释放内存）
worker.terminate();

    // worker.js（Worker 线程）
self.onmessage = (e) => {
  const { type, data } = e.data;
  
  if (type === 'SORT') {
    // 执行耗时操作（不会阻塞主线程）
    const sorted = data.sort((a, b) => a - b);
    
    // 返回结果
    self.postMessage({ result: sorted });
  }
};

// Worker 中可使用的 API：
// setTimeout/setInterval, fetch, XMLHttpRequest, IndexedDB, Cache API
// 不可使用：DOM, window, document, alert, localStorage（可间接通过主线程操作）

    Inline Worker（避免单独文件）：

    4. Service Worker

    Service Worker 是运行在浏览器背后的独立线程，可代理网络请求、管理缓存，是实现 PWA（渐进式 Web 应用）的核心技术。

    核心特性：

    • 离线访问：缓存资源，无网络时也能访问
    • 网络代理：拦截请求，决定从缓存还是网络获取
    • 后台同步：网络恢复后自动发送失败请求
    • 推送通知：接收服务器推送消息
    • 独立生命周期：与页面无关，可后台运行

    生命周期：

    注册(register) → 安装(install) → 激活(activate) → 闲置/终止
                     ↓                    ↓
                  缓存静态资源          清理旧缓存

    基本用法：

    // main.js：注册 Service Worker
if ('serviceWorker' in navigator) {
  window.addEventListener('load', () => {
    navigator.serviceWorker.register('/sw.js')
      .then(registration => {
        console.log('SW 注册成功:', registration.scope);
      })
      .catch(err => {
        console.log('SW 注册失败:', err);
      });
  });
}

    // sw.js：Service Worker 脚本
const CACHE_NAME = 'app-v1';
const STATIC_ASSETS = [
  '/',
  '/index.html',
  '/css/main.css',
  '/js/main.js',
  '/icon.png'
];

// 安装阶段：缓存静态资源
self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(CACHE_NAME)
      .then(cache => cache.addAll(STATIC_ASSETS))
      .then(() => self.skipWaiting()) // 立即激活新 SW
  );
});

// 激活阶段：清理旧缓存
self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches.keys().then(cacheNames => {
      return Promise.all(
        cacheNames
          .filter(name => name !== CACHE_NAME)
          .map(name => caches.delete(name))
      );
    }).then(() => self.clients.claim()) // 立即控制所有页面
  );
});

// 拦截网络请求
self.addEventListener('fetch', (event) => {
  const { request } = event;
  
  // 策略1：缓存优先（Cache First）- 适合静态资源
  event.respondWith(
    caches.match(request).then(response => {
      // 命中缓存则返回，否则请求网络并缓存
      return response || fetch(request).then(fetchResponse => {
        return caches.open(CACHE_NAME).then(cache => {
          cache.put(request, fetchResponse.clone());
          return fetchResponse;
        });
      });
    })
  );
});

    常见缓存策略：

    策略 流程 适用场景
    Cache First 先查缓存，无则请求网络并缓存 静态资源（CSS/JS/图片）
    Network First 先请求网络，失败则读缓存 实时数据（API 请求）
    Stale While Revalidate 立即返回缓存，同时后台更新 新闻、内容页面（兼顾速度和新鲜度）
    Network Only 只请求网络 敏感数据、必须实时的请求
    Cache Only 只读缓存 离线页面

    Stale While Revalidate 实现：

    self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.open(CACHE_NAME).then(cache => {
      return cache.match(event.request).then(cachedResponse => {
        // 发起网络请求（无论缓存是否命中）
        const fetchPromise = fetch(event.request).then(networkResponse => {
          // 更新缓存
          cache.put(event.request, networkResponse.clone());
          return networkResponse;
        }).catch(() => cachedResponse); // 网络失败回退缓存
        
        // 立即返回缓存（如果有），否则等待网络
        return cachedResponse || fetchPromise;
      });
    })
  );
});

    # 12. 怎么理解回流和重绘

    回流（Reflow / 重排）：

    • 当元素的大小、位置发生变化，浏览器需要重新计算布局
    • 触发条件：元素尺寸变化、内容变化、添加删除元素、窗口大小改变、读取 offset/scroll/client 等属性

    重绘（Repaint）：

    • 当元素的样式发生变化但不影响布局时，浏览器重新绘制元素
    • 触发条件：颜色、背景色、可见性（visibility）等变化

    关系：

    • 回流一定会触发重绘，重绘不一定会触发回流
    • 优化：批量修改样式、使用 className 代替逐条修改、使用文档碎片（DocumentFragment）、避免频繁读取布局属性

    # 15. 深拷贝和浅拷贝，怎么实现一个深拷贝说思路

    浅拷贝：只复制一层，深层对象共享引用

    • Object.assign()、Array.prototype.slice()、展开运算符 {...obj}

    深拷贝：完全复制，新旧对象互不影响

    实现思路：

    1. JSON 方法（最简单，但有局限）：JSON.parse(JSON.stringify(obj))

      • 缺点：丢失函数、undefined、Date、RegExp、循环引用、Symbol

    2. 递归实现（详细注释版）：

    /**
 * 深拷贝函数 - 递归克隆对象/数组，处理循环引用和特殊类型
 * @param {any} obj - 待克隆的数据
 * @param {WeakMap} hash - 缓存已克隆的对象，解决循环引用问题
 * @returns {any} 克隆后的新数据
 */
function deepClone(obj, hash = new WeakMap()) {
  // 【基准条件 1】处理 null 或原始类型（string/number/boolean/undefined/symbol/bigint）
  // 原始类型直接返回，无需克隆
  if (obj === null || typeof obj !== 'object') {
    return obj;
  }
  
  // 【特殊对象 1】处理 Date 对象
  // Date 存储的是时间戳，创建新实例并传入原时间戳
  if (obj instanceof Date) {
    return new Date(obj);
  }
  
  // 【特殊对象 2】处理 RegExp 正则对象
  // 通过原正则的 source（正则文本）和 flags（标志位）创建新实例
  if (obj instanceof RegExp) {
    return new RegExp(obj.source, obj.flags);
  }
  
  // 【循环引用处理】检查 WeakMap 中是否已克隆过该对象
  // 如果 obj 已经在 hash 中，说明出现了循环引用，直接返回之前创建的克隆对象
  // 这是避免无限递归导致栈溢出的关键
  if (hash.has(obj)) {
    return hash.get(obj);
  }
  
  // 【创建克隆对象】使用原对象的构造函数创建新实例
  // obj.constructor 指向创建 obj 的构造函数（Object/Array/自定义类）
  // 这样可以保留对象的原型链和自定义类类型
  const cloneObj = new obj.constructor();
  
  // 【建立引用映射】在递归前先存入 WeakMap
  // 原因：如果对象内部有循环引用，递归时会先查到这个映射，不会重复克隆
  // WeakMap 的键是对象，值是克隆后的新对象
  hash.set(obj, cloneObj);
  
  // 【递归克隆属性】遍历对象自身的可枚举属性
  for (let key in obj) {
    // hasOwnProperty 过滤掉原型链上的属性，只克隆对象自身的属性
    if (obj.hasOwnProperty(key)) {
      // 递归调用：对每个属性值进行深拷贝
      // 将 hash 传递下去，保证同一个遍历树中共享缓存
      cloneObj[key] = deepClone(obj[key], hash);
    }
  }
  
  // 返回克隆完成的新对象
  return cloneObj;
}

    函数执行流程图解：

    deepClone(objA)
  ├─ 检查 objA 是否在 hash 中？→ 否
  ├─ 创建 cloneA，hash.set(objA, cloneA)
  ├─ 遍历 objA 的属性
  │   ├─ key1: 原始值 → 直接返回
  │   ├─ key2: 引用 objB
  │   │   └─ deepClone(objB)
  │   │       ├─ 检查 objB 是否在 hash 中？→ 否
  │   │       ├─ 创建 cloneB，hash.set(objB, cloneB)
  │   │       └─ 遍历 objB 的属性...
  │   └─ key3: 引用 objA（循环引用！）
  │       └─ deepClone(objA)
  │           ├─ 检查 objA 是否在 hash 中？→ 是！
  │           └─ 返回 hash.get(objA) → cloneA（避免无限递归）
  └─ 返回 cloneA

    关键设计决策解析：

    代码设计 原因 如果不用会怎样
    new obj.constructor() 保留原型链和自定义类类型 数组会被克隆成普通对象，instanceof Array 失效
    hash = new WeakMap() 缓存已克隆对象，识别循环引用 循环引用导致无限递归，栈溢出（Stack Overflow）
    先 hash.set 再递归 对象 A 引用 B，B 又引用 A 时能正确识别 先递归再 set 无法处理双向循环引用
    hasOwnProperty 不克隆原型链上的方法 克隆了不必要的原型方法，污染结果
    单独处理 Date/RegExp 这类对象不能直接遍历属性克隆 丢失时间值或正则标志位

    局限性：

    • 不处理函数（函数克隆有争议，通常深拷贝不复制函数）
    • 不处理 Map、Set、WeakMap、WeakSet
    • 不处理 Symbol 作为键的属性
    • 不处理不可枚举属性
    • 生产环境推荐 lodash.cloneDeep（处理上述所有边界情况）

    # 【补充】WeakMap 详解

    为什么用 WeakMap 而不是 Map 或普通对象？

    1. WeakMap 的核心特性

    特性 WeakMap Map 普通对象
    键的类型 只能是对象 任意类型 只能是字符串/Symbol
    垃圾回收 键是弱引用，不影响 GC 键是强引用，阻止 GC 强引用
    可遍历 ❌ 不可遍历（无 size/keys/values） ✅ 可遍历 ✅ 可遍历
    性能 适合对象→值的映射 通用场景 简单场景

    2. "弱引用"是什么意思？

    弱引用意味着：如果 WeakMap 的键对象在其他地方不再被引用，垃圾回收器可以直接回收它，不需要先从 WeakMap 中删除。

    // 示例：对比 Map 和 WeakMap 的垃圾回收行为
let user = { name: 'Tom' };

// Map：强引用
const map = new Map();
map.set(user, 'user data');
user = null;  // 虽然 user 变量不再引用对象
// 但 Map 仍然强引用该对象 → 对象不会被垃圾回收 → 内存泄漏风险

// WeakMap：弱引用
const weakMap = new WeakMap();
weakMap.set(user, 'user data');
user = null;  // 没有其他强引用 → 垃圾回收器直接回收对象
// WeakMap 中对应的键值对也会自动消失（不可访问）

    3. 为什么深拷贝必须用 WeakMap？

    深拷贝过程中，hash 的键是源对象，值是克隆后的新对象。整个深拷贝是一个临时操作，完成后 hash 就不再需要了。

    如果用 Map：

    • 深拷贝结束后，hash 仍然持有所有源对象的强引用
    • 即使外部不再使用源对象，Map 中的引用阻止垃圾回收
    • 对于大对象，这会造成显著的内存泄漏

    用 WeakMap：

    • 深拷贝完成后，函数退出，hash 局部变量不再被引用
    • WeakMap 中的键如果没有其他强引用，会被垃圾回收
    • 不会阻止内存释放，零泄漏风险

    4. 为什么 WeakMap 不可遍历？

    因为键可能被垃圾回收随时消失，遍历结果是不确定的。

    const wm = new WeakMap();
let obj = { id: 1 };
wm.set(obj, 'data');

// 以下方法都不存在：
wm.size;        // undefined
wm.keys();      // TypeError
wm.values();    // TypeError
wm.entries();   // TypeError
wm.forEach();   // TypeError

// 唯一可用的方法：
wm.get(obj);    // 'data'
wm.has(obj);    // true
wm.delete(obj); // true

    5. WeakMap 的其他应用场景

    场景 A：为第三方对象添加私有数据（不污染原对象）

    const privateData = new WeakMap();

class User {
  constructor(name, password) {
    // 公开属性
    this.name = name;
    // 私有数据存储在 WeakMap 中，外部无法直接访问
    privateData.set(this, { password, loginCount: 0 });
  }
  
  login() {
    const data = privateData.get(this);
    data.loginCount++;
    console.log(`${this.name} 登录了 ${data.loginCount} 次`);
  }
}

const u = new User('Tom', 'secret');
console.log(u.name);        // 'Tom'
console.log(u.password);    // undefined（真正的私有）
u.login();                  // Tom 登录了 1 次

    场景 B：DOM 元素关联数据（元素移除后自动释放）

    const elementData = new WeakMap();

function attachData(element, data) {
  elementData.set(element, data);
}

const button = document.getElementById('btn');
attachData(button, { clickCount: 0, lastClick: null });

// 当按钮从 DOM 中移除且没有 JS 引用时
// elementData 中的对应条目会被自动垃圾回收，无需手动清理

    场景 C：缓存函数计算结果（避免内存无限增长）

    const cache = new WeakMap();

function computeExpensive(obj) {
  if (cache.has(obj)) {
    return cache.get(obj);  // 命中缓存
  }
  
  const result = /* 复杂计算 */ obj.data * 2;
  cache.set(obj, result);
  
  return result;
}

// 当 obj 不再被使用时，缓存条目自动释放
// 不会像普通 Map 那样无限累积缓存

    6. WeakMap vs WeakSet

    特性 WeakMap WeakSet
    存储内容 键值对（对象→任意值） 只有键（对象集合）
    典型方法 set/get/has/delete add/has/delete
    用途 对象关联数据 标记对象是否被处理过
    // WeakSet：标记对象是否已处理
const processed = new WeakSet();

function process(obj) {
  if (processed.has(obj)) {
    console.log('已处理过，跳过');
    return;
  }
  
  // ... 处理逻辑
  processed.add(obj);  // 标记为已处理
}

    总结：WeakMap 是 JavaScript 内存管理的利器，在需要"给对象附加临时数据且不干扰垃圾回收"的场景下（如深拷贝缓存、私有属性、DOM 元数据），它是不可替代的选择。

    # 16. 内存泄露是什么，怎么导致的

    内存泄露：程序中已分配的内存由于某种原因未释放或无法释放，造成系统内存浪费，可能导致程序变慢或崩溃。

    常见原因：

    1. 意外全局变量：未声明直接使用变量
    2. 闭包：内部函数持有外部函数变量的引用
    3. 定时器未清理：setInterval 未 clearInterval
    4. DOM 引用：移除 DOM 元素但 JS 中仍保留引用
    5. 事件监听未移除：addEventListener 未 removeEventListener
    6. console.log：开发环境打印大对象

    排查方法：

    • Chrome DevTools → Performance → Memory → Heap Snapshot
    • 对比前后快照，查看 retained size

    # 17. Promise 介绍

    Promise 是异步编程的一种解决方案，用于解决回调地狱问题。

    三种状态：

    • pending（进行中）→ fulfilled（已成功）/ rejected（已失败）
    • 状态一旦改变，不可再次改变

    常用方法：

    • Promise.resolve() / Promise.reject()
    • Promise.all()：全部成功才成功，一个失败就失败
    • Promise.race()：返回最快完成的
    • Promise.allSettled()：等待全部完成，无论成功失败
    • Promise.any()：任一成功即成功（ES2021）

    注意：

    • Promise 是微任务，优先于宏任务执行
    • 链式调用中，then 返回新的 Promise

    # 2. 如何实现一个 new

    new 运算符做了以下事情：

    1. 创建一个空对象
    2. 将对象的原型链接到构造函数的 prototype
    3. 将构造函数中的 this 绑定到这个对象
    4. 执行构造函数代码
    5. 返回这个对象（除非构造函数显式返回对象）
    function myNew(constructor, ...args) {
  // 1. 创建空对象，原型指向 constructor.prototype
  const obj = Object.create(constructor.prototype);
  // 2. 执行构造函数，绑定 this
  const result = constructor.apply(obj, args);
  // 3. 如果构造函数返回对象则返回该对象，否则返回新对象
  return result instanceof Object ? result : obj;
}

// 测试
function Person(name) {
  this.name = name;
}
const p = myNew(Person, 'Tom');
console.log(p.name); // Tom
console.log(p instanceof Person); // true

class EventBus {
  private events: Map<string, Function[]> = new Map();
  
  on(event: string, callback: Function) {
    if (!this.events.has(event)) this.events.set(event, []);
    this.events.get(event)!.push(callback);

    this.events.get(event)?.forEach(cb => cb(data));

    # 8. 跨域是什么？怎么解决？jsonp 方式是如何做的

    跨域（CORS）：浏览器的同源策略限制，协议、域名、端口任一不同即为跨域。

    解决方案：

    1. CORS（跨域资源共享）：服务端设置响应头
      Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET, POST
    2. JSONP：利用 <script> 标签不受同源策略限制
    3. 代理服务器：开发环境用 webpack devServer proxy，生产环境用 Nginx
    4. postMessage：窗口间通信
    5. WebSocket：不受同源策略限制

    JSONP 原理：

    1. 前端定义回调函数 handleData(data)
    2. 创建 <script> 标签，src 带上回调函数名 ?callback=handleData
    3. 服务端返回 handleData({"name": "Tom"})
    4. 脚本加载后执行回调函数
    function jsonp(url, callbackName) {
  return new Promise((resolve) => {
    const script = document.createElement('script');
    window[callbackName] = (data) => {
      resolve(data);
      document.body.removeChild(script);
    };
    script.src = `${url}?callback=${callbackName}`;
    document.body.appendChild(script);
  });
}

    JSONP 缺点：只支持 GET 请求，存在 XSS 安全隐患

    # 11. 了解 axios 的原理吗？怎么实现的

    Axios 是基于 Promise 的 HTTP 客户端，同时支持浏览器和 Node.js。

    核心原理：

    1. 浏览器端：基于 XMLHttpRequest
    2. Node.js 端：基于 http 模块

    核心功能实现：

    • 拦截器：使用数组存储请求/响应拦截器，promise 链式调用
    • 取消请求：基于 XMLHttpRequest 的 abort() 或 CancelToken
    • 并发请求：axios.all() 封装 Promise.all()
    • 自动转换 JSON：默认 transformRequest / transformResponse

    简单实现：

    class Axios {
  request(config) {
    return new Promise((resolve, reject) => {
      const xhr = new XMLHttpRequest();
      xhr.open(config.method || 'get', config.url);
      xhr.onreadystatechange = () => {
        if (xhr.readyState === 4) {
          if (xhr.status >= 200 && xhr.status < 300) {
            resolve(xhr.response);
          } else {
            reject(xhr.statusText);
          }
        }
      };
      xhr.send(config.data);
    });
  }
}

    # 6. websocket 的底层协议

    WebSocket 是基于 TCP 的全双工通信协议，在 HTTP 协议上升级而来。

    连接过程：

    1. 客户端发送 HTTP 请求，带上升级头：

    3. 连接建立，之后使用 ws:// 或 wss:// 通信

    特点：

    • 全双工：客户端和服务端可同时发送消息
    • 低延迟：建立连接后无需重复握手
    • 轻量头部：数据帧头部仅 2-14 字节
    • 支持二进制：可传输文本和二进制数据

    # 7. Vue2 和 Vue3 的 tree shaking 的区别

    Tree Shaking：消除未使用代码的优化技术，依赖 ES Module 的静态结构。

    Vue 2：

    • 使用 UMD 和 IIFE 构建，难以 tree shaking
    • 全局 API 挂载在 Vue 构造函数上（Vue.set、Vue.nextTick）
    • 即使未使用也会被打包

    Vue 3：

    • 全面采用 ES Module 构建
    • API 改为具名导出（import { nextTick } from 'vue'）
    • 未使用的 API 不会被打包
    • 按需引入，包体积显著减小（运行时约 10KB gzipped）

    Vue 3 的优化：

    • 编译时优化：静态提升、PatchFlag 标记动态节点
    • 源码采用 Monorepo，各模块独立发布

    # 14. 虚拟 DOM 的理解

    虚拟 DOM（Virtual DOM）：

    • 用 JavaScript 对象描述真实 DOM 结构的轻量级表示
    • 是真实 DOM 的抽象，与平台无关（可用于浏览器、Native、小程序等）

    工作流程：

    1. 用 JS 对象构建虚拟 DOM 树
    2. 状态变化时生成新的虚拟 DOM
    3. 通过 Diff 算法对比新旧虚拟 DOM
    4. 找出最小变更集（Patch）
    5. 批量更新真实 DOM

    优点：

    • 减少直接操作真实 DOM 的次数（操作 DOM 是昂贵的）
    • 跨平台（React Native、Weex）
    • 批量更新，减少重排重绘

    缺点：

    • 首次渲染不如直接操作 DOM 快
    • 需要额外的内存存储虚拟 DOM
    • Diff 算法也有计算开销

    Vue 的优化：

    • 编译时优化，标记动态节点（PatchFlag），减少比较范围
    • 静态提升：静态节点只创建一次

    # 【补充】Diff 算法原理详解

    1. 为什么需要 Diff 算法

    状态变化后生成新的虚拟 DOM 树，如果直接遍历两棵树逐个节点对比，时间复杂度是 O(n³)（树形结构对比的经典算法复杂度），这对于前端渲染来说是不可接受的。

    Diff 算法通过三个核心假设将复杂度优化到 O(n)：

    1. 同层比较：只对比同一层的节点，不跨层级比较
    2. 类型不同则直接替换：如果节点类型不同（如 div 变成 span），直接销毁旧树、创建新树
    3. 通过 key 标识同一组子节点：列表渲染时，用 key 判断节点是否相同，实现复用

    2. Diff 算法执行流程（以 Vue 2 的双端 Diff 为例）

    对比两个虚拟 DOM 树的流程：

    新 VNode 树                     旧 VNode 树
    div                            div
   /   \                          /   \
  p     span        Diff         p     div   ← 类型不同，直接替换
  |      |          ===>         |      |
 text   text                    text   text

    同层级节点的四种对比情况：

    场景 处理方式 示例
    新旧都是文本节点 直接替换文本内容 <div>旧</div> → <div>新</div>
    新旧节点类型不同 销毁旧节点，创建并挂载新节点 <div> → <span>
    新旧节点类型相同 对比属性差异，复用 DOM 节点，递归对比子节点 class="a" → class="b"
    新旧都是数组（子节点列表） 进入子节点 Diff（最复杂） 列表增删改顺序

    3. 子节点 Diff 的核心：双端比较算法

    当新旧子节点都是数组时（如 v-for 渲染的列表），Diff 算法采用双端比较策略：

    依次采用四种策略：

    - 旧头 - 新头，复用节点，双指针++
- 旧尾-新尾，复用节点，双指针--
- 旧头-新尾，复用节点，oldStart元素移动到oldEnd后，oldStart位置设置为undefinded，双指针内收
- 旧尾-新头，复用节点，，oldEnd元素移动到oldStart后，oldEnd位置设置为undefinded，双指针内收
- 在就数组遍历寻找该节点，找到则移动该元素到就旧数组头部（找不到在旧数组尾部创建）
    • 循环结束
      • 新数组有剩余，批量创建
      • 旧数组有剩余，批量删除
    // 伪代码：双端 Diff 核心逻辑
function updateChildren(parentElm, oldCh, newCh) {
  let oldStartIdx = 0;            // 旧数组头指针
  let oldEndIdx = oldCh.length - 1; // 旧数组尾指针
  let newStartIdx = 0;            // 新数组头指针
  let newEndIdx = newCh.length - 1; // 新数组尾指针
  
  let oldStartVnode = oldCh[0];
  let oldEndVnode = oldCh[oldEndIdx];
  let newStartVnode = newCh[0];
  let newEndVnode = newCh[newEndIdx];
  
  while (oldStartIdx <= oldEndIdx && newStartIdx <= newEndIdx) {
    // 四种命中策略（按优先级排序）
    
    // 策略 1：旧头 === 新头（最常用，如尾部追加）
    if (sameVnode(oldStartVnode, newStartVnode)) {
      patchVnode(oldStartVnode, newStartVnode); // 递归对比属性和子节点
      oldStartVnode = oldCh[++oldStartIdx];
      newStartVnode = newCh[++newStartIdx];
    }
    // 策略 2：旧尾 === 新尾（头部插入场景）
    else if (sameVnode(oldEndVnode, newEndVnode)) {
      patchVnode(oldEndVnode, newEndVnode);
      oldEndVnode = oldCh[--oldEndIdx];
      newEndVnode = newCh[--newEndIdx];
    }
    // 策略 3：旧头 === 新尾（移动旧头到尾部）
    else if (sameVnode(oldStartVnode, newEndVnode)) {
      patchVnode(oldStartVnode, newEndVnode);
      parentElm.insertBefore(oldStartVnode.elm, oldEndVnode.elm.nextSibling);
      oldStartVnode = oldCh[++oldStartIdx];
      newEndVnode = newCh[--newEndIdx];
    }
    // 策略 4：旧尾 === 新头（移动旧尾到头部）
    else if (sameVnode(oldEndVnode, newStartVnode)) {
      patchVnode(oldEndVnode, newStartVnode);
      parentElm.insertBefore(oldEndVnode.elm, oldStartVnode.elm);
      oldEndVnode = oldCh[--oldEndIdx];
      newStartVnode = newCh[++newStartIdx];
    }
    // 策略 5：以上都不命中，遍历旧数组查找可复用节点
    else {
      const idxInOld = findIdxInOld(newStartVnode, oldCh, oldStartIdx, oldEndIdx);
      if (idxInOld === -1) {
        // 新节点在旧数组中不存在，创建新 DOM
        createElm(newStartVnode, parentElm, oldStartVnode.elm);
      } else {
        // 找到可复用节点，移动到头部
        const vnodeToMove = oldCh[idxInOld];
        patchVnode(vnodeToMove, newStartVnode);
        parentElm.insertBefore(vnodeToMove.elm, oldStartVnode.elm);
        oldCh[idxInOld] = undefined; // 标记为已处理
      }
      newStartVnode = newCh[++newStartIdx];
    }
  }
  
  // 处理剩余节点
  if (oldStartIdx > oldEndIdx) {
    // 新数组有剩余，批量创建
    addVnodes(parentElm, newCh, newStartIdx, newEndIdx);
  } else if (newStartIdx > newEndIdx) {
    // 旧数组有剩余，批量删除
    removeVnodes(parentElm, oldCh, oldStartIdx, oldEndIdx);
  }
}

// 判断两个节点是否相同（复用 DOM 的前提）
function sameVnode(a, b) {
  return (
    a.key === b.key &&           // key 必须相同
    a.tag === b.tag &&           // 标签类型相同（div/span/p）
    a.isComment === b.isComment && // 同为注释节点
    isDef(a.data) === isDef(b.data) // 都有/都没有属性数据
  );
}

    四种命中策略图解：

    旧子节点: [A, B, C, D]      新子节点: [B, A, C, E]
            ↑  ↑                      ↑  ↑
         头指针 尾指针               头指针 尾指针

第 1 轮:
  策略 1: oldStart(A) vs newStart(B) → key 不同，不命中
  策略 2: oldEnd(D) vs newEnd(E) → key 不同，不命中
  策略 3: oldStart(A) vs newEnd(E) → key 不同，不命中
  策略 4: oldEnd(D) vs newStart(B) → key 不同，不命中
  策略 5: 在旧数组中查找 newStart(B)，找到索引 1
         → 移动旧 B 到头部，旧数组标记为 [A, undefined, C, D]
         新头指针右移

旧子节点: [A, undefined, C, D]  新子节点: [B, A, C, E]
                ↑    ↑                   ↑     ↑
             头指针  尾指针            头指针   尾指针

第 2 轮:
  策略 1: oldStart(A) vs newStart(A) → key 相同，命中！
         → 复用 A 的 DOM，递归 patch，双头指针右移
         ...（继续循环直到结束）

    为什么优先用双端比较而不是简单遍历？

    • 实际业务中，列表操作多为尾部追加、头部插入、逆序
    • 双端比较能在 O(n) 内处理这些场景
    • 简单遍历 + 查找需要 O(n²) 复杂度

    4. Key 的作用与重要性

    key 是 Diff 算法的"身份证"：

    <!-- 没有 key：Vue 只能按位置对比 -->
<div v-for="item in list">
  {{ item.name }}
</div>

<!-- 有 key：Vue 按 key 查找复用节点 -->
<div v-for="item in list" :key="item.id">
  {{ item.name }}
</div>

    用 index 作为 key 的问题：

    // 原始列表
[{ id: 1, name: 'A' }, { id: 2, name: 'B' }]

// 在头部插入新项
[{ id: 3, name: 'C' }, { id: 1, name: 'A' }, { id: 2, name: 'B' }]

// 如果用 index 作为 key：
// 旧: [0-A, 1-B]  新: [0-C, 1-A, 2-B]
// Diff 会认为：0 位置 A→C（更新），1 位置 B→A（更新），2 位置新增 B
// 结果：3 次 DOM 操作，且输入框等状态错乱

// 如果用 id 作为 key：
// Diff 识别出：id=3 是新增，id=1 和 id=2 只是位置变化
// 结果：1 次插入 + 2 次移动，输入框状态正确保留

    key 的最佳实践：

    • 使用唯一且稳定的标识（数据库主键、UUID）
    • 避免使用 index（除非列表只增不删不改顺序）
    • 避免使用 Math.random()（每次渲染 key 都变，导致全部重建）

    5. Vue 3 的 Diff 优化

    Vue 2 的问题：

    • 每个组件更新时，需要遍历整个组件的虚拟 DOM 树
    • 即使大部分内容是静态的，也要参与 Diff

    Vue 3 的优化策略：

    优化手段 原理 效果
    静态提升（Hoist Static） 编译时将静态节点提升到渲染函数外部 静态节点只创建一次，跳过 Diff
    PatchFlag（动态标记） 编译时标记节点的动态部分（文本、class、style 等） Diff 时只对比标记的部分
    Block Tree 将动态节点收集到数组，Diff 只遍历动态节点 跳过大量静态节点的比较

    PatchFlag 示例：

    // 模板
// <div class="static">
//   <p>静态文本</p>
//   <p>{{ dynamicText }}</p>
//   <p :class="activeClass">动态 class</p>
// </div>

// Vue 3 编译后的渲染函数（简化）
function render(_ctx) {
  // 静态节点被提升，只创建一次
  const _hoisted_1 = createStaticVNode("<p>静态文本</p>", 1);
  
  return openBlock(), createElementBlock("div", { class: "static" }, [
    _hoisted_1,  // 静态节点，直接复用
    createElementVNode("p", null, _ctx.dynamicText, 1 /* TEXT */),  // 只对比文本
    createElementVNode("p", {
      class: normalizeClass(_ctx.activeClass)
    }, null, 2 /* CLASS */)  // 只对比 class
  ]);
}

    PatchFlag 类型枚举：

    • 1 (TEXT)：动态文本内容
    • 2 (CLASS)：动态 class 绑定
    • 4 (STYLE)：动态 style 绑定
    • 8 (PROPS)：动态非 class/style 属性
    • 16 (FULL_PROPS)：有动态 key 属性，需要完整 Diff
    • 32 (HYDRATE_EVENTS)：需要客户端注水事件
    • 64 (STABLE_FRAGMENT)：子节点顺序不变的 Fragment
    • 128 (KEYED_FRAGMENT)：有 key 或部分子节点有 key
    • 256 (UNKEYED_FRAGMENT)：无 key 的子节点
    • 512 (NEED_PATCH)：组件需要强制更新
    • 1024 (DYNAMIC_SLOTS)：动态插槽
    • 2048 (DEV_ROOT_FRAGMENT)：开发环境根片段
    • -1 (HOISTED)：已提升的静态节点，跳过 Diff
    • -2 (BAIL)：退出优化模式，采用完整 Diff

    6. React 的 Diff 算法差异

    特性 Vue 2 Vue 3 React
    对比策略 双端比较 双端比较 + Block Tree 单链表遍历（从左到右）
    key 作用 查找复用节点 查找复用节点 + 快速路径 查找复用节点
    列表 Diff O(n) O(n)，跳过静态节点 O(n)，但需要 key
    编译优化 无 PatchFlag + 静态提升 无（依赖开发者 memo）
    子节点对比 同时对比头尾 同时对比头尾 只从左到右对比

    React 的 Reconciliation（协调）：

    React 使用单链表遍历（而非双端）：

    1. 遍历新列表，在旧列表中查找相同 key 的节点
    2. 找到则复用并移动，找不到则创建
    3. 遍历完成后，旧列表中未匹配的节点删除

    React 18 引入了 Automatic Batching 和 Concurrent Rendering，可以中断和恢复 Diff 过程，提升用户体验。

    # 21. 为什么 data 属性是一个函数而不是一个对象

    Vue 组件中的 data 必须是函数：

    // 正确
export default {
  data() {
    return { count: 0 };
  }
};

// 错误（会导致数据共享）
export default {
  data: { count: 0 }
};

    原因：

    • Vue 组件可能被多次复用（多个实例）
    • 如果 data 是对象，所有实例会共享同一个引用
    • 一个实例修改数据，会影响其他实例
    • data 作为函数，每次返回新对象，保证各实例数据独立

    根实例例外：

    • new Vue({ data: { count: 0 } }) 可以是对象，因为根实例只有一个

    # 6. useCallback 使用过没？

    useCallback：缓存函数引用，避免子组件不必要的重渲染。

    const memoizedCallback = useCallback(
  () => doSomething(a, b),
  [a, b]
);

    使用场景：

    1. 函数作为 prop 传递给子组件，且子组件做了 React.memo 优化
    2. 函数作为依赖项传给其他 Hook（如 useEffect）

    注意：

    • 不是所有函数都需要 useCallback，有开销
    • 只有当函数被缓存的依赖项消费时才有价值
    • React.memo + useCallback 常配合使用

    useMemo vs useCallback：

    • useMemo：缓存计算结果（值）
    • useCallback：缓存函数本身

    # 7. 函数组件和类组件处理重复渲染有什么区别？

    类组件：

    • shouldComponentUpdate(nextProps, nextState)：手动控制是否更新
    • PureComponent：浅比较 props 和 state
    • React.memo（函数组件）的类组件版本

    函数组件：

    • React.memo(Component, areEqual)：类似 PureComponent
    • useMemo：缓存计算值
    • useCallback：缓存函数
    • 优化思路：保证引用稳定（props、state、context）

    重复渲染原因：

    1. 父组件更新导致子组件更新（即使 props 没变）
    2. State 变化（即使值相同，如 setState({})）
    3. Context 变化（所有消费者重渲染）

    # 9. 数据什么时候定义在组件里面，什么时候定义在状态管理里面？

    定义位置 适用场景
    组件 State 局部状态，仅当前组件使用（如表单输入、弹窗显隐）
    状态管理 全局共享状态，多个组件需要访问（如用户信息、主题、购物车）

    判断标准：

    1. 作用范围：只有当前组件用 → State；跨组件共享 → Store
    2. 持久化需求：需要持久化或恢复 → Store
    3. 数据复杂度：简单状态 → State；复杂关联数据 → Store
    4. 服务端数据：从 API 获取且多处使用 → Store（配合缓存策略）

    # 10. 方法什么时候写在父组件中，什么时候写在子组件中？

    位置 场景
    父组件 涉及多个子组件协作、操作父组件状态、需要状态提升时
    子组件 仅操作子组件内部状态、封装可复用逻辑、组件自包含时

    原则：

    • 单一职责：子组件只处理自己的逻辑
    • 状态就近：数据在哪里使用，逻辑就放在哪里
    • 复用性：可复用的逻辑封装在子组件或自定义 Hook 中
    • 性能：频繁更新的状态尽量下沉，减少不必要的父组件渲染

    # 6. forEach 循环和 for 循环哪个性能高？forEach 循环可以中断吗？

    性能对比：

    循环方式 性能 特点
    for 最快 最基础，无函数调用开销
    for...of 较快 可迭代对象，有迭代器开销
    forEach 较慢 函数回调，每次迭代都是函数调用
    map/filter 较慢 创建新数组，有额外开销

    结论：普通 for 循环性能最好，但差距在大多数场景可忽略。

    forEach 能否中断：

    • 不能使用 break、continue、return 中断整个循环
    • 只能使用 return 跳过当前迭代（类似 continue）
    • 想中断需抛异常（不推荐）：
    try {
  arr.forEach(item => {
    if (item === target) throw new Error('break');
  });
} catch (e) {}

    可中断的替代方案：

    • for / for...of + break
    • some() / every()：返回 true/false 可中断
